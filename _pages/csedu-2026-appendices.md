# APPENDICES
## A.1 Evaluation Grid

Evaluation rubric used in the assessment process (by both the instructor and the LLM-based system). The rubric is organized into four categories (first column), each associated with a percentage weight. Each category includes one or more specific indicators. For every indicator, a corresponding guiding question is provided (third column), and responses are evaluated according to predefined performance levels. 

| Category | Indicator | Question | Levels (0–5) |
|----------|----------|----------|--------------|
| **Correctness (weight: 60%)** | Functional Completeness | Does the application correctly implement the basic and advanced tasks described in the assignment? | 0: Application does not correctly implement the required tasks.<br>1: Many basic tasks are missing or incorrect.<br>2: Some basic tasks are incomplete or partially incorrect.<br>3: All basic tasks are correctly implemented.<br>4: All basic tasks and most advanced tasks are correctly implemented.<br>5: All basic and advanced tasks are correctly implemented. |
| | Robustness | Does the implementation correctly handle edge cases, errors, and abnormal conditions? | 0: No handling of edge cases or errors.<br>1: Error handling almost absent.<br>2: Minimal or incomplete error handling.<br>3: Main cases handled but some checks missing.<br>4: Most edge cases handled correctly.<br>5: All edge cases and errors identified and handled correctly. |
| | Design Autonomy | Does the solution include autonomous design choices or improvements not explicitly required by the assignment? | 0: Solution copied or nearly identical to examples or other submissions.<br>1: Mechanical solution replicating examples without meaningful adaptation.<br>2: Only the minimal required solution with rigid design.<br>3: Standard approach aligned with the assignment without extensions.<br>4: At least one autonomous design choice improving the project.<br>5: Multiple motivated design choices improving quality, flexibility, or robustness. |
| | Specification Adherence | Were the submission instructions (format, naming, required files, execution environment) respected? | 0: Submission not compliant or not evaluable.<br>1: Severe issues complicating evaluation.<br>2: Several instructions not respected.<br>3: Some formal inaccuracies but the project is evaluable.<br>4: Minor non-critical formal inaccuracies.<br>5: All submission instructions respected. |
| **Maintainability (weight: 20%)** | Modularity | Is the code divided into modules or functions with well-defined responsibilities? | 0: Completely monolithic code.<br>1: Nearly monolithic code.<br>2: Limited modularity; functions perform multiple tasks.<br>3: Sufficient modularity but improvable.<br>4: Good modularity with rare violations of the single-responsibility principle.<br>5: Highly modular code with clearly separated responsibilities. |
| | Maintainability | Can the code be extended or modified without introducing significant errors? | 0: Code cannot be modified without rewriting.<br>1: Modifications extremely difficult.<br>2: Modifications complex and risky.<br>3: Modifications possible but require care.<br>4: Modifications possible with moderate effort.<br>5: Extensions and modifications are simple and safe. |
| **Readability (weight: 10%)** | Naming | Are variable, function, and class names clear and meaningful? | 0: Names incomprehensible or absent.<br>1: Misleading or confusing names.<br>2: Unclear or inconsistent names in several places.<br>3: Understandable but sometimes generic names.<br>4: Generally clear names with rare ambiguities.<br>5: Names always clear, consistent, and self-explanatory. |
| | Formatting | Do indentation, spacing, and coding style improve readability? | 0: No readable formatting.<br>1: Severely inadequate formatting.<br>2: Disorganized formatting.<br>3: Acceptable but inconsistent formatting.<br>4: Clear formatting with minor imperfections.<br>5: Impeccable and consistent formatting. |
| **Documentation (weight: 10%)** | Documentation | Do comments help understand the structure and behavior of the code? | 0: No comments at all.<br>1: Misleading or unnecessary comments.<br>2: Rare or not useful comments.<br>3: Comments present but limited.<br>4: Generally useful comments.<br>5: Clear, useful, and well-distributed comments. |
